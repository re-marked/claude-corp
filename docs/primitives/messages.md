# Messages

The communication record. Every message in the corporation is a single JSON line appended to a `messages.jsonl` file inside a [[channels|channel]] folder. JSONL (JSON Lines) — one complete JSON object per line, newline-delimited.

```
~/.agentcorp/corp-name/{channel-path}/messages.jsonl
```

## Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | string (ULID) | Unique, time-sortable identifier. Generated by the sender. |
| `channelId` | string | ID of the [[channels\|channel]] this message belongs to. Matches the channel registry in `channels.json`. |
| `senderId` | string | [[members\|Member]] ID of the author. Both users and agents populate this. |
| `threadId` | string or null | ID of the parent message for threaded replies. Null for top-level messages. |
| `content` | string | The message body. Plain text or markdown. |
| `kind` | enum | Message type: `text`, `system`, `task_event`. See [[#Kinds]]. |
| `mentions` | string[] | List of member IDs explicitly mentioned in `content` via `@displayName`. Extracted at write time. |
| `metadata` | object or null | Arbitrary key-value data. Used by `task_event` messages to carry structured payloads (task ID, status transitions, etc.). |
| `timestamp` | ISO 8601 | When the message was created. |

## Kinds

### text

Standard conversational message. Sent by users via the TUI or by agents via filesystem writes. This is the default and most common kind.

### system

Infrastructure announcements generated by the daemon or lifecycle hooks. Examples:

- "Cleo has joined #general"
- "Project website created"
- "Heartbeat cycle 47 complete — 3 agents active"

System messages have a null `senderId` or use a reserved `system` sender ID.

### task_event

Structured events about [[tasks]] lifecycle changes. The `metadata` field carries the payload:

```json
{"id":"01J...","channelId":"01J...","senderId":"01J...","threadId":null,"content":"Task 'Design landing page' assigned to Cleo","kind":"task_event","mentions":["01J_CLEO"],"metadata":{"taskId":"01J_TASK","transition":"pending->assigned","assignedTo":"01J_CLEO"},"timestamp":"2026-03-12T14:30:00Z"}
```

Task events are posted to both the relevant project/team channel and the system `#tasks` channel.

## Write Semantics

### Append-only

Messages are only ever appended. No edits, no deletes. The JSONL file is a strict append log. This makes writes simple, concurrent-safe (with atomic append), and perfectly git-diffable.

### Atomic writes

Every message write is a single `fs.appendFileSync` (or equivalent atomic append) of one JSON line followed by a newline character. This guarantees that concurrent writers (multiple agents, the user, the daemon) never produce corrupted partial lines.

```
{...message JSON}\n
```

No trailing comma. No wrapping array. Just lines.

### Who writes

Both the TUI and agents write directly to the JSONL file:

- **User (TUI)**: The user types a message, the TUI process resolves mentions, generates the ULID, and appends the line.
- **Agent**: The agent process constructs the JSON object and appends it to the file at the path it knows from its channel membership.
- **Daemon**: Writes `system` kind messages for lifecycle events.

There is no broker, no queue, no server in between. The filesystem *is* the message bus.

### Daemon routing

The daemon watches all active `messages.jsonl` files via `fs.watch`. When a new line appears:

1. Parse the last appended line.
2. Resolve `mentions` — look up each mentioned member in `members.json`.
3. For each mentioned agent that is `active`, forward the message content to the agent's local port.
4. If no explicit mentions in a broadcast [[channels|channel]], let each agent decide whether to respond based on its own heuristics.

The daemon does not store or transform messages. It only routes.

## Threading

Threads are flat: a message with a non-null `threadId` is a reply to the message with that ID. There is no nesting beyond one level. Thread replies live in the same `messages.jsonl` file as top-level messages — they are interleaved chronologically. The TUI groups them visually.

To load a thread: grep the JSONL file for all lines where `threadId` matches the parent message ID.

## Querying

JSONL is grep-friendly by design:

```bash
# All messages from a specific sender
grep '"senderId":"01J_CLEO"' messages.jsonl

# All task events
grep '"kind":"task_event"' messages.jsonl

# All messages mentioning a member
grep '"01J_MARK"' messages.jsonl | grep '"mentions"'

# Last 20 messages
tail -20 messages.jsonl
```

For the TUI, messages are read by tailing the file and parsing each line. For history, read the full file and parse line by line. ULIDs are time-sortable, so chronological order matches file order.

## Git Tracking

Every `messages.jsonl` file is git-tracked. Commits happen periodically (on heartbeat cycles or after bursts of activity). This means:

- Full conversation history is versioned and auditable.
- `git diff` shows exactly what was said between any two points in time.
- `git log --follow` on a channel's JSONL file shows the complete timeline.
- Revert is possible but destructive to conversation continuity — use with caution.
